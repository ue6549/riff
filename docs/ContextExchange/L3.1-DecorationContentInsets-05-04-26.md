## What Was Built

L3.1: Decoration contentInsets API — `sectionBackgroundContentInsets` on `ListLayoutDelegate` and `GridLayoutDelegate`.

Mirrors `NSCollectionLayoutDecorationItem.contentInsets`. Positive values shrink the sectionBackground frame inward; negative values expand outward. Applied at C++ frame emission time so windowing queries and ShadowNode positions use the inset-adjusted coordinates.

## API

```typescript
// In layout delegate:
layout={list({
  sectionBackground: true,
  sectionBackgroundContentInsets: { top: -8, bottom: -8 },  // jacket effect
})}

layout={grid({
  sectionBackground: true,
  sectionBackgroundContentInsets: { top: -60, left: -16, right: -16 },  // overlap-above
})}
```

The `decorationRenderers` renderer function receives the already-inset-adjusted frame — no JS-level transform needed.

## Implementation

### C++ (pure geometry — no new C++ types needed)

**`ListLayout.h` / `GridLayout.h`** — 4 new double fields on params structs:
```
sectionBackgroundInsetTop, sectionBackgroundInsetBottom,
sectionBackgroundInsetLeft, sectionBackgroundInsetRight
```

**`ListLayout.cpp` / `GridLayout.cpp`** — insets applied at sectionBackground emission:
- Both `computeSection` and `computeSectionFromCache` updated (both emit sectionBackground)
- Horizontal list: left/right apply to the scroll axis (X), top/bottom to cross axis (Y)
- Vertical list/grid: standard visual coordinates
- Insets applied with formula: `x += left, y += top, width -= left+right, height -= top+bottom`

**`paramsFromJSI`** in both files — parse 4 new fields from JS object.

### TypeScript

**`protocol.ts`** — `sectionBackgroundContentInsets?: { top?, bottom?, left?, right? }` added to both `ListLayoutDelegate` and `GridLayoutDelegate`.

**`list.ts` / `grid.ts`** — pass `sectionBackgroundInsetTop/Bottom/Left/Right` to C++ params from delegate.

**`list.ts` fingerprint** — inset values included so cache clears when insets change.

**`LayoutsTab.tsx`** — list demo now uses `sectionBackgroundContentInsets: { top: -8, bottom: -8 }` as a jacket-effect example.

## Key Design Decision

Insets live in the **layout delegate** (C++ param pipeline), not in `decorationRenderers`. This is correct because:
1. The layout engine must know the final frame to emit it correctly for windowing
2. `decorationRenderers` is a pure render concern — it just receives whatever frame C++ emits
3. Mirrors NSCollectionLayoutDecorationItem architecture exactly
